TWO GOTCHAS

Hai điều dễ nhầm

WIDE-COLUMN vs COLUMNAR OLAP
Tên na ná, mục đích ngược nhau. Cassandra (wide-column) tối ưu ghi/đọc theo partition key — OLTP. ClickHouse / DuckDB (columnar OLAP) tối ưu scan + aggregate — analytics.
THE LINE IS BLURRING
PostgreSQL nuốt dần nhiều loại: JSONB (document), pgvector (vector), TimescaleDB (time-series), PostGIS (geo). Nhiều khi chỉ cần một extension, không cần DB mới.

DEMO 1 · RACE CONDITION

House contention

100 user · 1 house trống · cùng một lúc — chỉ 1 người được thắng

3 APPROACHES

Chống race condition — 3 cách

Naive PostgreSQL

read rồi update, không lock
  • Nhiều request cùng pass check NULL
  • Nhiều user cùng "thắng" → oversubscribed

PostgreSQL FOR UPDATE

lock row trong transaction
  • Đúng hoàn toàn — chỉ 1 thắng
  • Spike lớn → xếp hàng sau row lock

Redis SET NX EX

atomic, tự expire 30'
  • Không lock, không queue — cực nhanh
  • Không phải truth — phải ghi Postgres sau
BENCHMARK · GUESS vs REAL

100 request cùng hit 1 house

KY VONGFOR UPDATE chậm nhất (~300ms) · naive lệch vài winner · Redis ~30ms nhanh nhất
APPROACHWINNERSLATENCYCORRECTNESS
Naive PostgreSQL~80~100msFAIL — oversubscribed
PostgreSQL FOR UPDATE1~22msCorrect
Redis SET NX1~3msCorrect
BUSTEDnaive ~80 winner (tệ hơn nghĩ), FOR UPDATE chỉ ~22ms — không phải chậm nhất. Đo, đừng đoán.
DO THAT · 100 CONCURRENT · LOCALHOST
RIGHT PATTERN

Kết hợp Redis + Postgres

Redis SET NXchặn race, coordination
↓ success
PostgreSQL UPDATEghi truth, audit log
↓ 30 phút sau
Redis TTL expire + cleanuptự giải phóng
Crash giữa hai bước? TTL là cái cứu — lock tự hết hạn, không kẹt vĩnh viễn. Và PostgreSQL là truth → độ lệch luôn bị chặn trên.
TAKEAWAY · DEMO 1
Redis → coordination ngắn hạn, atomic, cực nhanh.
PostgreSQL → final truth, audit log, không mất khi restart. Không để Redis làm truth cuối cùng · không để Postgres làm coordination dưới spike.
  • Scale bottleneck: Postgres hot row · Redis hot key — nằm ở access pattern, không phải lượng data.

WHEN TO PICK

Khi nào chọn gì

  • PG thuần — quan hệ phức tạp, schema ổn định → users, wallets, transactions
  • PG JSONB — muốn flexible, không thêm DB → hero_config, building_config
  • MongoDB — document là access pattern chính, object lớn, ít JOIN
Flexible schema không tự động = MongoDB. JSONB là middle ground mạnh: flexibility + ACID + một DB duy nhất.

CONDITION · TAKEAWAY

"Không đụng OLTP" — với một điều kiện

  • Đưa battle_logs sang OLAP: export Parquet định kỳ · CDC · read-replica
  • Khâu sync cũng tốn tài nguyên → chạy off-peak / trên replica, đừng scan primary giờ cao điểm
Tách OLAP khỏi OLTP là quyết định kiến trúc, không phải tối ưu performance — tách cả compute lẫn đường lấy data.
PITFALLS · REAL WORLD

Khi kiến trúc đẹp đổ vỡ

Team không bắt đầu với kiến trúc đẹp — mà với deadline, rồi accumulate những quyết định nhỏ tưởng vô hại

PITFALLS · 1-2

1 · Redis làm source of truth

Vấn đề: in-memory, restart mất data; có MULTI/EXEC + Lua nhưng không rollback / isolation như SQL.
Dấu hiệu: viết script "sync Redis về Postgres" chạy định kỳ.

2 · MongoDB nhưng tư duy SQL

Vấn đề: $lookup không có query planner như JOIN, không FK — chậm hơn SQL JOIN.
Dấu hiệu: pipeline $lookup 3–4 stage khắp nơi.
PITFALLS · 3-4

3 · Duplicate data không kiểm soát

Vấn đề: embed/denormalize (document hoặc JSONB) → mất single source of truth, 2 nơi ra 2 kết quả.
Dấu hiệu: "data hiển thị khác nhau tùy chỗ".

4 · SQL ôm luôn analytics

Vấn đề: thêm index cho analytics → OLTP write chậm; hai workload triệt tiêu nhau.
Dấu hiệu: báo cáo schedule chạy 2 giờ sáng.
PITFALLS · 5-6

5 · Hiểu sai scalability

Vấn đề: nhiều user ≠ cùng bottleneck; contention scale ≠ data volume; schema scale ≠ throughput.
Dấu hiệu: "thêm read replica" nhưng vẫn nghẽn write contention.

6 · Vội polyglot, thêm DB quá sớm

Vấn đề: mỗi DB thêm = thêm vận hành + bài toán consistency, trước khi có scale thật.
Dấu hiệu: mỗi DB một service nhưng chỉ chứa vài nghìn record.
PITFALLS · RECAP
Không có database nào tệ tuyệt đối — vấn đề là dùng sai storage cho sai workload. Redis làm truth → mất data · Mongo tư duy SQL → chậm hơn SQL · Duplicate → không biết đâu đúng · Analytics đè OLTP → cả hai chậm · Hiểu sai bottleneck → sai vấn đề · Vội polyglot → phức tạp trước khi cần

STEP 2 · 4 QUESTIONS

4 câu hỏi trước khi chọn DB

1 · Workload là gì?

Đọc nhiều, point query → SQL + index
Ghi nhiều, append-only → wide-column / partition
Aggregate scan → OLAP riêng

2 · Consistency cần gì?

Mất tiền / NFT → ACID, Postgres
Rank lệch vài giây → eventual, Redis
Report hơi cũ → batch sync, DuckDB

3 · Access pattern?

Theo ID + JOIN → relational SQL
Nguyên object nested → document / JSONB
Top N liên tục → Sorted Set

4 · Bottleneck ở đâu?

Data volume → partition / OLAP
Write contention → Redis coordination
Schema đổi nhanh → JSONB / document

OPEN DISCUSSION

Câu hỏi gợi mở

  • Hệ thống hiện tại có chỗ nào analytics chạy trên OLTP không? Báo cáo có schedule lúc 2 giờ sáng?
  • "Source of truth cho balance / inventory là đâu" — trả lời ngay được, hay phải họp team?
  • Lần gần nhất thêm field vào entity chính — mất bao lâu từ quyết định đến deploy?
  • Lần hệ thống chậm gần nhất — bottleneck thật ở đâu: data volume, write contention, query nặng, hay connection pool?
